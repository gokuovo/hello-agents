# 第一章 初识智能体 - 学习笔记

来源：[docs/chapter1/第一章 初识智能体.md](../../docs/chapter1/第一章%20初识智能体.md)

## 本章学习目标

- 能用一句话说明什么是智能体：感知环境、自主决策、采取行动、达成目标。
- 能讲清楚传统智能体与 LLM 智能体的区别。
- 能记住 PEAS、Agent Loop、Thought-Action-Observation 三个核心框架。
- 能区分 Workflow 和 Agent 的适用场景。
- 能用面试语言解释：工具调用、规划、观察反馈、循环终止、风险控制。

## 1.1 什么是智能体？

### 1.1.1 传统视角下的智能体

智能体的经典定义：任何能够通过传感器感知环境，并自主地通过执行器采取行动，以达成特定目标的实体。

必须记住的四个要素：

| 要素 | 含义 | 例子 |
| --- | --- | --- |
| Environment 环境 | 智能体所处的外部世界 | 道路交通、金融市场、软件系统 |
| Sensors 传感器 | 获取环境信息的入口 | 摄像头、麦克风、雷达、API 返回数据 |
| Actuators 执行器 | 改变环境状态的手段 | 机械臂、方向盘、代码执行、服务调用 |
| Autonomy 自主性 | 基于感知和内部状态独立决策 | 不只是执行固定脚本，而是选择下一步行动 |

传统智能体的演进路径：

| 类型 | 核心机制 | 回答的问题 | 典型例子 | 局限或重点 |
| --- | --- | --- | --- | --- |
| 简单反射智能体 | 条件-动作规则 | 当前感知对应哪个动作？ | 自动恒温器 | 无记忆、无上下文 |
| 基于模型的反射智能体 | 内部世界模型 | 世界现在是什么样？ | 自动驾驶对遮挡车辆的估计 | 有初级记忆，但目标性有限 |
| 基于目标的智能体 | 搜索与规划 | 我该怎么做才能达成目标？ | GPS 路径规划、A* 搜索 | 关注是否达成目标 |
| 基于效用的智能体 | 效用函数与权衡 | 哪个行动结果最令人满意？ | 同时考虑时间、成本、舒适度的路线选择 | 能处理多目标冲突 |
| 学习型智能体 | 经验反馈与策略更新 | 如何通过经验变得更好？ | AlphaGo Zero、强化学习系统 | 依赖奖励、反馈和大量交互 |

面试容易问：

- “智能体是不是等于一个很强的模型？”不是。模型只是智能体的大脑或决策模块之一，智能体还需要环境、感知、行动、反馈和目标。
- “一台超级计算机是不是智能体？”通常不是。算力本身不等于智能体，关键看它是否能感知环境、自治决策并采取行动。
- “基于目标和基于效用的区别？”目标型关注能否到达目标状态；效用型关注多个可行状态中哪个更优。

### 1.1.2 大语言模型驱动的新范式

LLM 智能体的变化不只是“把模型换大了”，而是构建方式变了：

| 对比项 | 传统智能体 | LLM 智能体 |
| --- | --- | --- |
| 能力来源 | 显式规则、人工知识、特定算法 | 预训练获得的隐式世界模型和语言推理能力 |
| 交互方式 | 多为结构化输入输出 | 能处理自然语言、高层级、模糊目标 |
| 行动方式 | 按预设逻辑执行 | 能规划、调用工具、根据反馈调整 |
| 适用任务 | 边界清晰、流程稳定 | 上下文复杂、需要动态推理和工具组合 |
| 风险 | 覆盖不到新情况时失效 | 可能幻觉、不可控、成本高、稳定性不足 |

旅行助手例子要记住三点：

- 规划与推理：把“规划一次旅行”拆成确认偏好、查信息、定行程、订票务等子任务。
- 工具使用：发现自己缺少实时信息时，调用天气、搜索、地图、订票等外部工具。
- 动态修正：把用户反馈当作新约束，例如预算不够就重新搜索。

面试表达模板：

> LLM Agent 的核心价值是把大模型的语言理解和推理能力，连接到外部工具与环境反馈中，让系统能够围绕目标进行多步规划、执行和修正。

### 1.1.3 智能体的类型

第一种分类：按内部决策架构。

- 反应式：当前输入直接映射到动作。
- 模型式：维护内部世界模型，应对部分可观察环境。
- 目标式：通过搜索、规划选择能达成目标的动作。
- 效用式：在多个目标和结果之间做权衡。
- 学习型：通过经验反馈持续改进策略。

第二种分类：按时间与反应性。

| 类型 | 特点 | 优点 | 缺点 | 例子 |
| --- | --- | --- | --- | --- |
| Reactive 反应式 | 几乎立即响应 | 快、成本低 | 短视、难做复杂规划 | 安全气囊、高频交易 |
| Deliberative 规划式 | 行动前深度思考 | 有远见、适合复杂目标 | 慢、计算成本高 | 棋类 AI、商业计划生成 |
| Hybrid 混合式 | 快速反应 + 长期规划 | 兼顾灵活性和目标性 | 设计复杂 | 现代 LLM Agent |

第三种分类：按知识表示。

| 范式 | 知识形式 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 符号主义 AI | 规则、逻辑、知识图谱 | 可解释、可追踪 | 脆弱，难覆盖真实世界所有例外 |
| 亚符号主义 AI | 神经网络中的统计模式 | 擅长感知和非结构化数据 | 黑箱，可能幻觉 |
| 神经符号主义 AI | 神经网络 + 结构化推理 | 兼顾模式识别与逻辑推理 | 工程实现更复杂 |

卡尼曼“双系统”类比：

- 系统 1：快速直觉，对应神经网络的模式识别。
- 系统 2：慢速推理，对应符号逻辑、规划和审慎决策。
- 现代 LLM Agent 常常像神经符号混合体：内部是神经网络，外部输出计划、工具调用、结构化步骤。

## 1.2 智能体的构成与运行原理

### 1.2.1 任务环境定义

PEAS 是描述任务环境的标准框架：

| 字母 | 全称 | 要回答的问题 | 旅行助手例子 |
| --- | --- | --- | --- |
| P | Performance 性能度量 | 怎样算做得好？ | 推荐准确、用户满意、成本低、行程合理 |
| E | Environment 环境 | 智能体处在哪些外部条件中？ | 用户偏好、天气、航班、酒店、景点、API |
| A | Actuators 执行器 | 智能体能做什么动作？ | 查询 API、搜索网页、生成建议、下单预订 |
| S | Sensors 传感器 | 智能体从哪里获得信息？ | 用户输入、API 返回、搜索结果、订单状态 |

LLM 智能体常见环境特性：

- 部分可观察：只能看到 API 或搜索返回的一部分信息。
- 随机性：机票、库存、天气、价格可能随时变化。
- 多智能体：其他用户、自动化脚本、平台调价系统也会改变环境。
- 序贯性：当前动作会影响后续状态。
- 动态性：智能体思考时，环境本身也可能变化。

面试容易问：

- “为什么要做 PEAS？”因为它能把一个模糊需求拆成可工程化的任务定义。
- “环境是部分可观察时怎么办？”需要记忆、探索、重复查询、置信度判断和异常处理。

### 1.2.2 智能体的运行机制

Agent Loop 是智能体的核心运行机制：

```text
感知 Perception
  -> 思考 Thought
      -> 规划 Planning
      -> 工具选择 Tool Selection
  -> 行动 Action
  -> 环境状态变化 State Change
  -> 观察 Observation
  -> 下一轮循环
```

关键理解：

- 智能体通常不是一次性完成任务，而是多轮推进。
- Observation 既可能来自用户，也可能来自上一步工具调用。
- Thought 阶段会更新任务理解，拆解子任务，选择工具和参数。
- Action 会通过执行器影响环境。
- 新的 Observation 会进入下一轮循环，形成闭环。

一句话记忆：

> Agent 不是“问一次答一次”，而是在环境反馈中不断感知、思考、行动、再观察。

### 1.2.3 智能体的感知与行动

工程上需要交互协议，让 LLM 输出可以被程序解析和执行。

本章使用的核心协议：

```text
Thought: 用户想知道北京的天气。我需要调用天气查询工具。
Action: get_weather("北京")
```

外部 Parser 会读取 `Action`，调用对应函数。工具执行后的结果会被包装成：

```text
Observation: 北京当前天气为晴，气温25摄氏度，微风。
```

然后把 Observation 送回 LLM，进入下一轮 Thought 和 Action。

工程重点：

- `Thought` 是模型的推理和计划。
- `Action` 是可执行指令，通常对应函数调用。
- `Observation` 是工具或环境返回的结果，要尽量简洁清晰。
- Parser 是 LLM 文本世界和真实工具世界之间的桥。
- 必须处理格式错误、未知工具、参数解析失败、工具调用失败等异常。

## 1.3 动手体验：5 分钟实现第一个智能体

### 1.3.1 准备工作

本节目标：实现一个智能旅行助手，完成“查询北京天气，然后根据天气推荐旅游景点”。

需要的组件：

- Python 依赖：`requests`、`tavily-python`、`openai`。
- System Prompt：规定角色、可用工具、输出格式、终止方式。
- 工具 1：`get_weather(city)`，通过 `wttr.in` 查询天气。
- 工具 2：`get_attraction(city, weather)`，通过 Tavily 搜索景点推荐。
- 工具注册表：`available_tools`，把工具名映射到真实函数。

System Prompt 的关键要求：

- 每次只输出一对 `Thought` 和 `Action`。
- `Action` 必须是一行，便于解析。
- 调用工具时使用 `function_name(arg_name="arg_value")`。
- 信息足够时使用 `Finish[最终答案]` 结束。

面试可讲：

> Prompt 在这里不是普通提示词，而是智能体的行为协议。它定义了模型如何思考、如何调用工具、何时结束任务。

### 1.3.2 接入大语言模型

本章实现了 `OpenAICompatibleClient`，用来调用兼容 OpenAI API 风格的模型服务。

需要配置：

- `API_KEY`
- `BASE_URL`
- `MODEL_ID`

为什么用 OpenAI-compatible 接口：

- 便于切换不同服务商。
- 既可接 OpenAI，也可接 Azure、Ollama、vLLM 等兼容服务。
- 降低智能体主循环与模型供应商之间的耦合。

### 1.3.3 执行行动循环

主循环做了这些事：

1. 保存用户请求到 `prompt_history`。
2. 拼接历史记录，构建本轮 Prompt。
3. 调用 LLM 生成 `Thought` 和 `Action`。
4. 截断多余输出，保证每轮只处理一对 Thought-Action。
5. 解析 `Action`。
6. 如果是 `Finish[...]`，输出最终答案并结束。
7. 如果是工具调用，从 `available_tools` 中找到函数并执行。
8. 把工具结果包装成 `Observation`，加入历史，进入下一轮。
9. 最多循环 5 次，避免无限循环。

这段代码的核心不是工具本身，而是“模型输出 -> 解析 -> 执行工具 -> 反馈观察”的闭环。

面试高频点：

- 为什么要限制最大循环次数？防止模型陷入重复调用、无法终止、成本失控或死循环。
- 为什么要有工具注册表？让模型只能调用被允许的工具，降低执行风险。
- 为什么要把工具结果转成 Observation？让模型用自然语言上下文继续推理。
- 正则解析有什么问题？对格式敏感，生产环境更适合结构化工具调用、JSON Schema 或框架级 parser。

### 1.3.4 运行案例分析

案例执行过程：

1. 第一轮：模型判断需要天气，调用 `get_weather(city="北京")`。
2. 第二轮：模型根据天气 Observation，调用 `get_attraction(city="北京", weather="Sunny")`。
3. 第三轮：模型综合天气和景点结果，使用 `Finish[...]` 给出最终答案。

这个小案例展示了四项 Agent 基本能力：

- 任务分解：把一个自然语言请求拆成多个步骤。
- 工具调用：知道什么时候需要外部信息。
- 上下文理解：能利用上一轮 Observation 继续推理。
- 结果合成：把多个工具结果整合成用户可读答案。

## 1.4 智能体应用的协作模式

### 1.4.1 作为开发者工具的智能体

这种模式下，智能体是增强开发者工作流的工具。它的定位不是完全替代开发者，而是自动化重复劳动、辅助编码、调试、解释和重构。

代表工具：

- GitHub Copilot：编辑器内补全和对话式编程。
- Claude Code：终端内执行代码编辑、测试、调试等任务。
- Trae：代码生成、优化和快速原型开发。
- Cursor：AI 原生代码编辑器，强调理解整个代码库上下文。

要记住的趋势：

> AI 正在从单点补全工具，进入软件开发全生命周期的人机协同工作流。

### 1.4.2 作为自主协作者的智能体

自主协作者模式的重点是“目标委托”。人类不再指挥每一步，而是给出高层级目标，由智能体独立规划、执行、反思和交付。

常见架构范式：

| 范式 | 说明 | 代表 |
| --- | --- | --- |
| 单智能体自主循环 | 一个通用 Agent 不断思考、规划、执行、反思 | AgentGPT、AutoGPT、BabyAGI |
| 多智能体协作 | 多个角色分工对话，共同完成复杂任务 | CAMEL、CrewAI、MetaGPT、AutoGen、AgentScope |
| 高级控制流架构 | 用状态图表达循环、分支、回溯、人工介入 | LangGraph |

面试表达：

> 自主协作者的关键变化，是从“命令-执行”变成“目标-委托”。系统需要具备任务拆解、执行监控、错误恢复和最终交付能力。

### 1.4.3 Workflow 和 Agent 的差异

核心区别：

> Workflow 是让 AI 按预设流程执行；Agent 是让 AI 围绕目标自主决策。

| 维度 | Workflow | Agent |
| --- | --- | --- |
| 底层逻辑 | 预定义流程、规则和分支 | 目标导向、动态规划和行动 |
| 自主性 | 低，按流程走 | 高，可选择步骤和工具 |
| 稳定性 | 高，结果可预测 | 较低，依赖模型和上下文 |
| 可解释性 | 强，流程清晰 | 需要日志、轨迹和约束辅助解释 |
| 适合场景 | 规则明确、合规要求高、流程稳定 | 信息不完整、任务开放、需要推理和适应 |
| 风险 | 灵活性不足 | 幻觉、越权调用、成本失控、不可重复 |

电商退款这类题的推荐答案思路：

- 简单低风险退款：用 Workflow，成本低、稳定、可审计。
- 复杂争议退款：用 Agent 辅助分析用户历史、商品情况、沟通记录。
- 最佳方案通常是混合方案：Workflow 负责硬规则和审批边界，Agent 负责信息整理、异常判断、建议生成，关键动作保留人工确认。

## 本章必须记住的 10 句话

1. 智能体的本质是“感知环境 -> 自主决策 -> 采取行动 -> 达成目标”。
2. 自主性是智能体区别于普通脚本的关键。
3. 传统智能体从反射、模型、目标、效用一路演进到学习型智能体。
4. LLM Agent 的新能力来自自然语言理解、隐式世界模型、规划能力和工具使用。
5. PEAS 用来定义任务环境：性能、环境、执行器、传感器。
6. Agent Loop 是智能体持续工作的闭环：感知、思考、行动、观察。
7. Thought-Action-Observation 是 LLM Agent 常见的工程交互协议。
8. 工具调用让 LLM 能接触实时信息和外部系统。
9. Workflow 适合稳定流程，Agent 适合开放目标。
10. 生产级 Agent 必须考虑循环终止、权限控制、错误处理、可观测性和安全边界。

## 面试高频问题与回答提纲

### 1. 什么是 Agent？

Agent 是能够感知环境、基于目标自主决策，并通过行动影响环境的系统。它通常包含感知模块、决策模块、工具或执行器、记忆或状态，以及反馈循环。

### 2. Agent 和普通程序有什么区别？

普通程序通常按固定逻辑执行，输入和输出边界清楚。Agent 更强调自主性和反馈闭环，会根据环境观察调整下一步行动。

### 3. Agent 和 Workflow 有什么区别？

Workflow 是预定义步骤，强调确定性和可审计；Agent 是目标驱动，强调动态规划和适应性。实际工程中常用混合模式：Workflow 管边界，Agent 管复杂判断。

### 4. PEAS 模型是什么？

PEAS 是 Performance、Environment、Actuators、Sensors，用来描述智能体任务环境。它能把“我要做一个智能助手”这种模糊需求拆成可设计、可评估的系统定义。

### 5. 什么是 Agent Loop？

Agent Loop 是感知、思考、行动、观察的循环。智能体通过不断获取反馈，修正计划并推进任务。

### 6. Thought-Action-Observation 有什么作用？

它把 LLM 的自然语言推理变成可执行流程。`Thought` 负责计划，`Action` 负责指令，`Observation` 负责把工具结果反馈给模型。

### 7. 为什么 LLM Agent 需要工具？

因为 LLM 本身无法可靠获取实时信息，也不能直接影响外部环境。工具让它能搜索、查数据库、调用 API、执行代码，从“会说”变成“能做”。

### 8. 为什么需要最大循环次数？

为了避免无限循环、重复调用、成本失控和任务迟迟不能结束。循环次数是 Agent 安全控制的一部分。

### 9. 反应式、规划式、混合式智能体怎么区分？

反应式速度快但短视；规划式能做长远计划但慢；混合式把快速反馈和长期规划结合起来。现代 LLM Agent 通常属于混合式。

### 10. 如何评价一个 Agent 的智能程度？

不能只看准确率，还要看任务成功率、工具调用正确率、鲁棒性、成本、延迟、可解释性、安全性、用户满意度和异常恢复能力。
